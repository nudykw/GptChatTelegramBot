# 🗄️ Посібник з міграцій бази даних

🇺🇸 [English](MIGRATIONS.md) | 🇺🇦 Українська

Проєкт підтримує **чотири** провайдери баз даних: **SQLite**, **PostgreSQL**, **MySQL** та **SQL Server**.
Оскільки кожен провайдер використовує власний SQL-діалект, міграції генеруються окремо для кожного з них і зберігаються в окремих папках у `DataBaseLayer/Migrations/`.

---

## Два варіанти застосунку

У рішенні є **два запускні застосунки**, що спільно використовують `DataBaseLayer`:

| Проєкт | Тип | Призначення |
|---|---|---|
| `TelegramBotApp/` | Консольний застосунок | Локальна розробка, просте розгортання |
| `TelegramBotWebApp/` | ASP.NET Core Web API | Docker, Webhook-режим, Prometheus-метрики, Swagger |

Обидва проєкти спільно використовують один **`DataBaseLayer`** і ті самі **міграції**. Скрипти міграцій за замовчуванням використовують `TelegramBotApp/` як startup-проєкт (він містить `appsettings.json` з усіма необхідними ключами). Можна також вказати `TelegramBotWebApp/` — результат ідентичний, адже схема БД визначається виключно в `DataBaseLayer`.

> [!NOTE]
> Міграції застосовуються **автоматично при запуску** в обох застосунках через `MigrateAsync()` / `MigrationConfigurator.ApplyMigrations()` — вручну команду `database update` у продакшені запускати ніколи не потрібно.

---

## Вимоги

Перед початком роботи з міграціями переконайтесь, що встановлено EF Core CLI:

```bash
dotnet tool restore
```

> [!NOTE]
> Інструмент оголошено у `dotnet-tools.json`, тому `dotnet tool restore` — все, що потрібно. Глобальна установка не потрібна.

---

## Як це працює

### Автоматичне застосування міграцій при запуску

Застосунок автоматично викликає `MigrateAsync()` під час запуску, тому **не потрібно** вручну виконувати `dotnet ef database update` у продакшені чи Docker-контейнері. Правильний провайдер обирається на основі `DB_PROVIDER` (з `.env` / `appsettings.json`), і застосовується лише відповідний набір міграцій.

### Мультипровайдерна архітектура

| Концепція | Деталь |
|---|---|
| **Папка міграцій** | `DataBaseLayer/Migrations/<Провайдер>/` |
| **Простір імен** | `DataBaseLayer.Migrations.<Провайдер>` |
| **Правило суфіксу** | Назва кожного класу міграції **повинна** закінчуватись на `_Sqlite`, `_Postgres`, `_MySql` або `_SqlServer` |
| **Ізоляція знімків** | Кожен провайдер зберігає власний `StoreContextModelSnapshot` як `.snapshot_saved` |
| **Маршрутизатор** | `DataBaseLayer/Internal/ProviderSpecificMigrationsAssembly.cs` завантажує правильні міграції під час виконання |

## 🚀 Перша міграція (початкове налаштування)

Виконайте один раз при першому налаштуванні проєкту або після клонування репозиторію. Створює початкову схему бази даних для **всіх чотирьох провайдерів** одночасно.

### За допомогою допоміжного скрипту *(рекомендовано)*

**Bash (Linux / macOS):**
```bash
./migr.sh add Initial
```

**PowerShell (Windows / кросплатформено):**
```powershell
./migr.ps1 add Initial
```

Скрипт виконає:
1. Перебере всі 4 провайдери по порядку: `Sqlite → PostgreSql → MySql → SqlServer`.
2. Для кожного провайдера виконає `dotnet ef migrations add Initial_<Суфікс>` з правильними `--output-dir` та `--namespace`.
3. Збереже згенерований `ModelSnapshot` для кожного провайдера, щоб наступні міграції могли будуватись на ньому.

Після завершення команди у `DataBaseLayer/Migrations/` з'являться чотири нові файли (по одному на провайдер):

```
DataBaseLayer/Migrations/
├── Sqlite/
│   └── 20240101000000_Initial_Sqlite.cs
├── Postgres/
│   └── 20240101000000_Initial_Postgres.cs
├── MySql/
│   └── 20240101000000_Initial_MySql.cs
└── SqlServer/
    └── 20240101000000_Initial_SqlServer.cs
```

> [!IMPORTANT]
> Міграції вже включені до репозиторію. Запускайте `add Initial` лише якщо починаєте з нуля (наприклад, видалили папку `Migrations/`).

## ➕ Наступні міграції (зміни схеми)

Кожного разу, коли ви змінюєте EF Core-сутність (додаєте властивість, перейменовуєте стовпець, додаєте нову таблицю тощо), потрібно створити нову міграцію для **всіх провайдерів**.

```bash
# Bash
./migr.sh add ОписоваНазва

# PowerShell
./migr.ps1 add ОписоваНазва
```

Замініть `ОписоваНазва` на коротке camelCase-опис змін, наприклад:

```bash
./migr.sh add AddUserLanguageField
./migr.sh add RenameBalanceColumn
./migr.sh add AddMessageIndexes
```

Скрипт автоматично додасть суфікс провайдера:
- `AddUserLanguageField_Sqlite`
- `AddUserLanguageField_Postgres`
- `AddUserLanguageField_MySql`
- `AddUserLanguageField_SqlServer`

> [!WARNING]
> **Ніколи** не додавайте міграцію лише для одного провайдера — всі чотири мають залишатись синхронізованими. Якщо потрібно швидко протестувати, однаково генеруйте для всіх провайдерів і локально застосовуйте лише потрібний.

## 🛠️ Застосування міграцій вручну

Міграції застосовуються автоматично при запуску, але ви також можете застосовувати їх вручну:

### Застосувати для поточного налаштованого провайдера
```bash
# Bash
./migr.sh update

# PowerShell
./migr.ps1 update
```

### Застосувати через EF CLI напряму (приклад для SQLite)
```bash
dotnet ef database update \
    --project DataBaseLayer/ \
    --startup-project TelegramBotApp/
```

### Застосувати для конкретного провайдера
Спочатку вкажіть `DB_PROVIDER` та `DB_CONNECTION_STRING` у `.env` / `appsettings.json`, після чого виконайте:
```bash
dotnet ef database update \
    --project DataBaseLayer/ \
    --startup-project TelegramBotApp/ \
    -- PostgreSql
```

> [!TIP]
> Якщо ви працюєте переважно з **веб-застосунком**, замініть `TelegramBotApp/` на `TelegramBotWebApp/` у будь-якій з команд вище. Результат ідентичний, адже схема БД визначається виключно в `DataBaseLayer`.

## 🗑️ Видалення останньої міграції

Використовуйте лише у тому випадку, якщо міграція **ще не застосована** до жодної реальної бази даних (наприклад, одразу після генерації з помилкою):

```bash
# Bash
./migr.sh remove

# PowerShell
./migr.ps1 remove
```

Це видаляє останню міграцію для всіх чотирьох провайдерів і відновлює їхні знімки до попереднього стану.

## ✏️ Ручне створення міграції (один провайдер)

Якщо потрібно додати міграцію лише для одного провайдера (не рекомендується окрім відлагодження):

```bash
dotnet ef migrations add MyChange_Sqlite \
    --project DataBaseLayer/ \
    --startup-project TelegramBotApp/ \
    --output-dir Migrations/Sqlite \
    --namespace DataBaseLayer.Migrations.Sqlite \
    -- Sqlite
```

Замініть `Sqlite` / `_Sqlite` на потрібний провайдер (`PostgreSql` / `_Postgres`, `MySql` / `_MySql`, `SqlServer` / `_SqlServer`).

---

## ⚠️ Важливі правила

1. **Завжди оновлюйте всі провайдери.** Будь-яка зміна моделі повинна генерувати міграції для всіх 4 провайдерів. Використовуйте допоміжні скрипти.
2. **Використовуйте описові camelCase-назви.** Скрипт додає суфікси автоматично — не включайте їх самостійно.
3. **Ніколи не перейменовуйте файли знімків вручну.** Якщо треба реорганізувати знімки, також оновіть `ProviderSpecificMigrationsAssembly.cs`.
4. **Фіксуйте міграції у репозиторії.** Міграції — частина вихідного коду, їх потрібно версіонувати разом із змінами моделі.
5. **Не змішуйте зміни схеми в одній міграції.** Одна логічна зміна = одна міграція. Це зберігає чистоту історії і спрощує відкат.

## 🔍 Усунення несправностей

| Проблема | Рішення |
|---|---|
| `No migrations found` при запуску | Виконайте `./migr.sh add Initial` для генерації початкової міграції |
| Помилка `Snapshot mismatch` | Видаліть застарілі файли `.snapshot_saved` та повторіть `add` з чистого стану |
| Міграція застосована до невірного провайдера | Перевірте `DB_PROVIDER` у `.env` / `appsettings.json` |
| EF CLI не знайдено | Виконайте `dotnet tool restore` у корені репозиторію |
| `Unable to create DbContext` під час міграції | Переконайтесь, що `TelegramBotApp/` компілюється і `appsettings.json` існує |
